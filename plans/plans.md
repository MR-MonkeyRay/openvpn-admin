设计并实现一个用于管理单台 Linux 服务器上已安装 OpenVPN 的 Web UI 管理端（面向管理员使用），要求轻量化、容器化、操作简单。请给出清晰的功能范围、系统架构与实现方案（包含前后端模块划分、数据模型、关键接口/流程与部署方式），并明确可验收的交付物。

## 1) 运行与集成范围
- 目标环境：单台 Linux 主机（OpenVPN 已安装并运行），Web UI 通过容器部署（Docker / Docker Compose）。
- Web UI 需要“管理主机上实际存在的 OpenVPN 配置与证书文件”，而不是仅管理一套独立的虚拟配置。
- 需要明确：Web UI 容器与宿主机 OpenVPN 的交互方式（例如挂载 `/etc/openvpn`、调用系统命令、或通过管理接口），以及所需的最小权限与安全边界。

## 2) 核心功能需求（必须）
### 2.1 OpenVPN Server 配置管理
- 读取并展示当前 Server 配置（如：监听端口/协议、网段、push 规则、DNS、路由、客户端配置目录 CCD、日志路径等）。
- 支持对关键配置项进行“表单化编辑 + 保存 + 配置校验 + 应用（reload/restart）”的流程。
- 变更必须支持：
  - 变更预览（diff）
  - 失败回滚（至少保留最近 N 次备份，可配置 N）
  - 操作审计记录（时间、操作者、变更内容摘要）

### 2.2 轻量化用户管理（用于发放客户端配置）
- 内置用户体系（非依赖外部 OAuth/LDAP），支持：
  - 创建/禁用/删除用户
  - 设置/重置用户名与密码（密码需安全存储：强哈希、不可明文）
  - 可选：用户到 OpenVPN 客户端证书的一对一或一对多关联规则（需说明采用哪种）
- 登录鉴权：Web UI 管理端登录（管理员账户）与普通用户（仅下载自己配置）的权限边界需明确。
- 最低权限模型：至少区分“管理员”和“普通用户/只读下载”。

### 2.3 按 Server 生成“客户端专属”配置
- 基于当前 OpenVPN Server 配置生成客户端 `.ovpn`（或同等形式）：
  - 每个用户/客户端实例拥有独立证书/密钥（或明确说明使用的认证方式：证书、用户名密码、或混合）
  - 可配置是否内联证书（inline）或以文件打包（例如 zip）
  - 支持配置项：remote 地址/端口/协议、cipher/auth、tls-crypt/tls-auth、redirect-gateway、DNS 等（以与 server 配置一致为原则）
- 提供客户端配置的生命周期管理：
  - 生成/下载
  - 吊销（revocation）与重新签发
  - 过期时间与续期策略（若支持）

### 2.4 证书与密钥管理（Server 与 Client）
- 展示并管理 Server 证书/CA/密钥（至少包含：路径、有效期、序列号、指纹、状态）。
- 支持：
  - 查看证书详情（解析 X.509 信息）
  - 生成/导入/替换 CA 与 Server 证书（需明确是否支持一键初始化 PKI）
  - 客户端证书签发、吊销、导出 CRL，并将 CRL 正确应用到 OpenVPN
- 所有敏感私钥的存储与访问策略必须明确（容器内/宿主机、权限、备份与导出限制）。

## 3) 非功能性要求（必须）
- 轻量化：说明目标资源占用（例如容器内存/CPU 的大致目标范围）与依赖最小化策略。
- 容器化交付：提供 `Dockerfile` 与 `docker-compose.yml`（或等效方式），可一键启动；明确需要挂载的宿主机目录与环境变量。
- 易用性：UI 页面结构需简洁（不超过 X 个一级菜单，X 由你建议并说明理由），核心流程（创建用户→生成配置→下载→吊销）不超过 3–5 步。
- 安全：必须包含
  - HTTPS/TLS 部署建议（可用反向代理或容器内 TLS）
  - CSRF/会话管理策略
  - 输入校验与命令执行安全（避免注入）
  - 操作审计与日志（敏感信息脱敏）
- 兼容性：明确支持的 OpenVPN 版本范围与配置目录的假设（例如 systemd service 名称、默认路径可配置）。
